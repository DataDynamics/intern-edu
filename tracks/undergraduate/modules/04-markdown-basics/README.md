# 모듈 04 — Markdown 기초

> **포커스**: 문서화, README/PR 작성, GitHub Flavored Markdown(GFM)
> **예상 기간**: 2~3일
> **선행 모듈**: 03 개발 환경 & 터미널

데이터 엔지니어의 하루를 들여다보면, 코드를 짜는 시간 못지않게 **글을 쓰는 시간**이 많습니다. 새로 만든 파이프라인이 무슨 일을 하는지 README에 설명하고, 동료의 코드를 리뷰하며 의견을 남기고, 왜 이렇게 고쳤는지 PR 설명에 적고, 분석 결과를 리포트로 정리합니다. 이 모든 글쓰기의 공통 언어가 바로 **Markdown**입니다. 지금 여러분이 읽고 있는 이 문서도, 앞으로 모든 모듈에서 만나게 될 문서도 전부 Markdown으로 쓰였습니다.

Markdown이 사랑받는 이유는 단순합니다. 복잡한 워드프로세서 없이 **평범한 텍스트에 약간의 기호만 더하면** 제목, 목록, 표, 코드가 보기 좋게 표현되기 때문입니다. 규칙이 적어서 5분이면 배우지만, 한번 익히면 평생 씁니다. 이 모듈의 목표는 단순히 문법을 외우는 것이 아니라, **읽는 사람을 배려한 문서를 쓰는 감각**을 기르는 것입니다.

---

## 🎯 이 모듈을 마치면

제목·강조·목록·링크·이미지를 자유롭게 쓰고, 코드 블록과 표로 정보를 정리하며, 체크리스트 같은 GitHub 확장 문법까지 활용해 **남이 읽기 좋은 README와 PR 설명**을 작성할 수 있게 됩니다. 그리고 작성한 문서를 미리보기로 확인하는 습관까지 갖추게 됩니다.

---

## 📚 본문

### 제목으로 글의 뼈대를 세운다

좋은 문서에는 위계가 있습니다. 가장 큰 주제가 있고, 그 아래에 작은 주제들이 가지를 칩니다. Markdown에서는 문장 맨 앞에 `#` 기호를 붙여 제목을 만드는데, `#`의 개수가 곧 제목의 깊이입니다. `#` 하나는 가장 큰 제목(제목 1), `##`는 그다음, 이런 식으로 `######`(제목 6)까지 내려갑니다.

```markdown
# 문서의 제목 (보통 한 문서에 하나)
## 큰 단원
### 그 아래 소단원
```

여기서 중요한 관습이 하나 있습니다. **제목 1(`#`)은 문서 전체를 대표하는 제목이므로 한 문서에 하나만** 두고, 그 아래 내용은 `##`부터 시작하는 것이 보기에도 좋고 검색·목차 생성에도 유리합니다. 또 `#`과 글자 사이에는 반드시 한 칸을 띄워야 합니다. `#제목`처럼 붙여 쓰면 제목으로 인식되지 않습니다.

### 강조 — 목소리의 높낮이를 주기

말에 강약이 있듯, 글에도 강조가 필요합니다. 정말 중요한 단어는 별표 두 개로 감싸 **굵게** 만들고, 가볍게 짚고 싶은 부분은 별표 하나로 *기울임*을 줍니다. 취소선(`~~`)은 "이건 더 이상 유효하지 않다"는 뉘앙스를 전할 때 씁니다.

```markdown
**굵게**      *기울임*      ~~취소선~~      `인라인 코드`
```

이 중에서 데이터 엔지니어가 가장 자주 쓰는 것은 의외로 **인라인 코드**(백틱 한 개로 감싸기)입니다. 문장 안에서 `파일명`, `함수명()`, `--옵션` 같은 기술 용어를 언급할 때 백틱으로 감싸면, 일반 문장과 코드가 시각적으로 구분되어 오해가 줄어듭니다. "config 파일을 고치세요"보다 "`config.yaml` 파일을 고치세요"가 훨씬 명확하지요.

### 목록 — 생각을 나란히 세우기

여러 항목을 나열할 때는 목록을 씁니다. 순서가 중요하지 않으면 `-`(하이픈)로 시작하는 **순서 없는 목록**을, 단계처럼 순서가 의미를 가지면 `1.`, `2.`로 시작하는 **순서 있는 목록**을 씁니다.

```markdown
- 사과
- 바나나
  - 제주산 바나나   ← 공백 두 칸(또는 네 칸)을 들여쓰면 하위 항목

1. 재료를 씻는다
2. 끓는 물에 넣는다
3. 5분 뒤 건진다
```

들여쓰기로 목록을 **중첩**할 수 있다는 점을 기억하세요. 큰 항목 아래 세부 항목을 단계적으로 늘어놓으면 복잡한 구조도 한눈에 들어옵니다. 재미있는 사실 하나는, 순서 있는 목록에서 숫자를 꼭 1, 2, 3으로 정확히 쓸 필요가 없다는 것입니다. 전부 `1.`로 써도 렌더링하면 자동으로 1, 2, 3이 매겨집니다. 그래서 중간에 항목을 추가해도 번호를 다시 매길 필요가 없습니다.

### 링크와 이미지 — 문서 밖으로 손을 뻗기

문서는 혼자 존재하지 않습니다. 다른 문서, 공식 사이트, 참고 자료로 독자를 안내해야 할 때가 많지요. 링크는 대괄호 안에 **보여줄 글자**를, 소괄호 안에 **주소**를 적습니다.

```markdown
[GitHub 공식 문서](https://docs.github.com/)
```

이미지는 링크 문법 앞에 느낌표(`!`) 하나만 더 붙이면 됩니다. 이때 대괄호 안의 글자는 화면에 보이는 게 아니라, 이미지가 깨졌을 때 대신 표시되거나 화면 낭독기가 읽어 주는 **대체 텍스트**입니다.

```markdown
![매출 추이 그래프](images/revenue.png)
```

링크 주소를 본문에 길게 늘어놓기 싫다면 **참조 방식**도 있습니다. 본문에는 짧은 이름표만 두고, 실제 주소는 문서 아래쪽에 모아 두는 방식입니다. 같은 링크를 여러 번 쓰거나 주소가 길 때 문서가 한결 깔끔해집니다.

```markdown
자세한 내용은 [공식 문서][docs]를 참고하세요.

[docs]: https://docs.github.com/ko
```

### 코드 블록 — 기술 문서의 심장

데이터 엔지니어의 문서에서 가장 중요한 요소를 하나만 꼽으라면 단연 **코드 블록**입니다. 명령어나 코드 여러 줄을 보여줄 때는 백틱 세 개(```` ``` ````)로 위아래를 감쌉니다. 이때 첫 백틱 뒤에 언어 이름을 적어 주면, 그 언어의 문법에 맞춰 색이 입혀져(syntax highlighting) 훨씬 읽기 좋아집니다.

````markdown
```python
def hello(name):
    return f"Hello, {name}!"
```
````

언어 이름은 `python`, `sql`, `bash`, `json`처럼 적습니다. 별것 아닌 것 같지만, 언어를 명시한 코드 블록과 그렇지 않은 코드 블록은 가독성 차이가 큽니다. 좋은 기술 문서일수록 이 한 단어를 빼먹지 않습니다.

### 인용문과 구분선 — 흐름에 쉼표 찍기

본문과 결이 다른 내용 — 팁, 주의사항, 인용 — 은 인용문으로 분리합니다. 문장 앞에 `>`를 붙이면 됩니다. 이 모듈 문서 곳곳의 `💡`, `⚠️`, `📌` 박스가 모두 인용문입니다. 긴 문서를 단원별로 시각적으로 끊고 싶을 때는 하이픈 세 개(`---`)로 가로 구분선을 그립니다.

```markdown
> 💡 팁: 인용문은 본문에서 잠깐 강조하고 싶은 내용을 담기에 좋습니다.

---
```

### 표 — 정보를 격자에 정리하기

비교할 항목이 여럿일 때는 표가 최고입니다. 세로 막대(`|`)로 칸을 나누고, **헤더 바로 아래에 `|---|` 구분선을 반드시** 넣어야 표로 인식됩니다. 이 구분선을 빼먹는 것이 초보자가 가장 자주 하는 실수입니다.

```markdown
| 단계 | 하는 일 | 도구 |
|------|---------|------|
| 수집 | 데이터를 가져온다 | API, 로그 |
| 저장 | 데이터를 쌓는다 | 데이터베이스 |
```

구분선에 콜론(`:`)을 더하면 칸의 정렬을 조절할 수도 있습니다. `:---`는 왼쪽, `:---:`는 가운데, `---:`는 오른쪽 정렬입니다. 숫자가 들어가는 칸을 오른쪽 정렬하면 자릿수가 맞아 읽기 편해집니다.

### 한 걸음 더 — 알아 두면 좋은 것들

문서를 쓰다 보면 사소하지만 자주 막히는 지점들이 있습니다. 몇 가지만 미리 알아 둡시다.

첫째, **줄바꿈**입니다. Markdown에서는 문장 끝에서 그냥 엔터를 한 번 치는 것만으로는 줄이 바뀌지 않습니다. 문단을 나누려면 **빈 줄을 하나** 넣어야 하고, 같은 문단 안에서 줄만 바꾸고 싶다면 줄 끝에 공백 두 칸을 두거나 `<br>`을 씁니다. "왜 내가 엔터를 쳤는데 한 줄로 붙어 나오지?" 하는 당황의 99%는 이 규칙 때문입니다.

둘째, **특수문자 그대로 쓰기**입니다. `*`나 `` ` `` 같은 기호를 강조가 아니라 글자 그대로 보여주고 싶다면, 앞에 백슬래시(`\`)를 붙여 탈출(escape)시킵니다. 예를 들어 `\*별표\*`라고 쓰면 *기울임*이 아니라 글자 그대로 \*별표\*가 나옵니다.

셋째, **방언(flavor)의 차이**입니다. Markdown에는 표준격인 CommonMark가 있고, GitHub이 여기에 표·체크리스트·자동 링크 등을 더한 **GitHub Flavored Markdown(GFM)**이 있습니다. 우리가 GitHub에서 쓰는 것은 GFM이며, 다음에 볼 체크리스트가 바로 GFM의 대표 기능입니다.

### 체크리스트 — 이 교육이 매일 쓰는 문법

GFM에는 작은 네모 칸이 달린 목록이 있습니다. 대괄호 사이에 공백을 두면 빈 칸(`- [ ]`), x를 넣으면 체크된 칸(`- [x]`)이 됩니다.

```markdown
- [x] GitHub 계정 만들기
- [ ] 첫 미니 프로젝트 완성하기
```

사실 여러분은 이 문법을 이미 매일 보고 있습니다. 모든 모듈 끝의 **`✅ 완료 기준`이 바로 이 체크리스트**입니다. 할 일을 적고 하나씩 채워 가는 이 단순한 도구는, GitHub Issue나 PR에서 작업 진행 상황을 공유할 때도 똑같이 쓰입니다.

### 좋은 README는 어떻게 생겼나

문법을 다 익혔다면, 이제 그것을 엮어 **좋은 문서**를 만들 차례입니다. README는 프로젝트를 처음 만난 사람을 위한 안내문입니다. 좋은 README는 대체로 비슷한 골격을 갖습니다. 맨 위에 프로젝트 이름과 한 줄 소개가 있고, 그 아래로 "어떻게 설치하는지", "어떻게 쓰는지", "예시는 무엇인지"가 차례로 이어집니다.

```markdown
# 프로젝트 이름
한 줄로 무엇을 하는 프로젝트인지 설명.

## 설치
## 사용법
## 예시
```

핵심은 **읽는 사람의 입장에서 순서를 짜는 것**입니다. 처음 온 사람이 가장 먼저 궁금해할 것부터 답해 주는 문서가 좋은 문서입니다.

### 미리보기 — 쓰면서 확인하기

Markdown은 기호로 쓰고 결과는 렌더링되어 보이므로, 쓰는 도중에 결과를 확인하는 습관이 중요합니다. VSCode에서는 `.md` 파일을 연 상태에서 `Cmd/Ctrl + Shift + V`를 누르면 미리보기 창이 열립니다. GitHub에 올린 파일은 자동으로 렌더링되어 보이니, 표나 코드 블록이 의도대로 나왔는지 꼭 눈으로 확인하세요.
